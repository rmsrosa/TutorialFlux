Flux.train!¶

  • O Flux.train! parte do mdulo Flux.Optimise.

  • Vejamos o cdigo do Flux.train!:

function train!(loss, ps, data, opt; cb = () -> ())
  ps = Params(ps)
  cb = runall(cb)
  @progress for d in data
    try
      gs = gradient(ps) do
        loss(batchmemaybe(d)...)
      end
      update!(opt, ps, gs)
      cb()
    catch ex
      if ex isa StopException
        break
      elseif ex isa SkipException
        continue
      else
        rethrow(ex)
      end
    end
  end
end

Detalhes¶

  • Params e gradient so importados do FluxML/Zygote.jl, que um pacote de diferenciao automtica.

  • Params um tipo composto que guarda quais parmetros e em que ordem devemos diferenciar uma funo

  • params, utilizado no treinamento visto anteriormente, define quais parmetros sero considerados para a diferenciao.

  • gradient calcula o gradiente de uma funo, em relao aos parmetros dados, via diferenciao automtica.

  • gradient usa backwards propagation (veja o uso de pullback(f, args...) na sua definio).

  • Observao: Podemos "pular" o Flux.train! e escrever mtodos de otimizao com outros mtodos de diferenciao, como ForwardDiff (forwards propagation) e ReverseDiff (backward mas mais genrico e no to eficiente quando o gradient no caso de redes neurais em questo). Leia mais sobre isso e sobre outros pacotes em JuliaDiff.

Atalho para uma sequência de treinos¶

  • comum vermos o uso de 
@epochs N train!(...)
  • Isso apenas um atalho para um loop com a exibio de poca a cada iterao:
for i=1:N
    train!(...)
end

Callbacks¶

  • O parmetro de callback pode ser muito til para

    • exibir informaes sobre determinados passos do loop de treinamento.
    • Registrar o custo/perda a cada passo para posterior exibio e avaliao.
    • afetar o treinamento de alguma forma:
      • interromper quando um determinada acurcia alcanada;
      • interromper quando um limite de iteraes ou de tempo alcanado;
      • trocar os parmetros correntes de busca caso o treinamente no esteja indo com muito sucesso;
      • substituir, acrescentar ou retirar determinados parmetros do processo de treinamento;
      • validar o treinamento de acordo com outra amostra de dados;
      • etc.

  • Vale lembrar que callbacks esto presentes em outros pacotes tambm, como DifferentialEquations, etc., para executar alguma instruo no meio de algum outro processo (e.g. resoluo de uma EDO).

  • O Callback uma keyword, ou seja, preciso passar como cb = funcao_de_callback, exceto que, no Julia 1.6, se a prpria funo tiver o nome da keyword, i.e. cb, no caso, e se usarmos ; para separar os argumentos do tipo keyword dos anteriores, ento podemos pass-la direto como Flux.train!(loss, ps, data, opt; cb).

  • Veja mais em Callback Helpers.

Inicialização das camadas¶

  • A inicializao das camadas um ponto importante do processo.

  • Isso particularmente importante em camadas convolucioais ou recorrentes.

  • Por default, Flux.jl inicializa essas camadas com o mtodo Flux.glorot_uniform.

  • Outros mtodos esto disponveis: glorot_normal, kalman_uniform, kalman_normal.

  • Esses mtodos de inicializao geram parmetros do tipo Float32! Por questes de otimizao para GPU (que so de 32-bits).

  • Em algumas situaes, caso a camada esteja como Float32, podemos ter erros (e.g. com RNN) ou avisos de "lentido no processamento" (e.g. com Conv) ao aplicamos a rede um vetor do tipo Float64.

  • possvel, no entanto, definir camadas com parmetros do tipo Float64 (ou outro qualquer) fornecendo diretamento os parmetros (e.g. m = Dense(rand(3,2)), ou um mtodo de inicializao (e.g. m = Dense(2,3, init=rand)).

  • Outra opo converter todos os parmetros de uma camada ou de uma rede m para Float64 com o comando m = fmap(f64, m). A funo f64 do Flux converte os parmetros para Float64 e a funo fmap, tambm do pacote FluxML/Functors.jl, mapeia todos os parmetros de volta para a camada/rede.

  • Mais sobre isso em Layer Initialization.

Parâmetros para otimização¶

  • O mtodo params pode exibir todos os parmetros de uma camada ou de uma rede

  • Mas a sua funo principal indicar, ao Flux.train! (ou, mais precisamente, ao Zygote.gradient), em relao a quais parmetros a funo de custo deve ser diferenciada e, assim, otimizada.

  • Podemos indicar todos os parmetros de uma rede m a serem otimizados passando ps = params(m) para o Flux.train!.

  • Podemos treinar apenas os neurnios de uma determinada camada, digamos, a segunda camada, com ps = params(m[2]).

  • Podemos treinar apenas alguns dos pesos de uma das camadas com ps = params(m[2].weight[1:2])

  • Aps selecionarmos alguns parmetros, podemos acrescentar e remover parmetros com os mtodos push! e delete!.

  • Por exemplo, se ps = params(m[1:3]), podemos acrescentar a quarta camada e remover parte da segunda camada com push!(ps, m[4]) e delete!(ps, m[2].weight[3:end]).

  • Veja mais sobre isso em Customising Parameter Collection for a Model.

Métodos de otimização¶

  • Uma das opes, em Flux.train!(loss, ps, data, opt, cb) o mtodo de otimizao opt.

  • No h opo default para o mtodo de otimizao. preciso escolher um.

  • H vrias opes.

  • Todos os mtodos esto implementados como diferentes "despachos" da funo Flux.Optimise.apply!, em src/optimise/optimisers.jl.

  • Essa funo apply! chamada no processo de atualizao do passo, dentro da funo update!, aps o gradiente ter sido calculado.

Treinando em lotes e o gradiente estocástico¶

  • No caderno anterior, fizemos exemplos com passos "completos", ou seja, levando-se em considerao todos os dados da amostra em cada "poca".

  • O mtodo de gradiente estocstico utiliza apenas um dado da amostra em cada passo, escolhido aleatoriamente, completando uma "poca" quando todos os dados so utilizados (no caso sem reposio) ou com algumas reutilizaes e outros esquecios (no caso com reposio).

  • E um caso intermedirio o em lotes (batch), em que a amostra dividida em grupos iguais escolhidos aleatoriamente e todos de um mesmo grupo so considerados a cada passo.

  • Essa metodologia pode ser aplicada, na verdade, em conjunto com qualquer mtodo de otimizao, no apenas o de gradiente descendente.

  • Para implementar isso, podemos fazer essa seleo manualmente ou usar o Flux.Data.DataLoader, que faz essa diviso dos dados em lotes do tamanho desejado.

Usando o Dataloader¶

  • O DataLoader nos poupa o trabalho de particionar os dados.

  • Alm dos dados, os outros argumentos para o mtodo construtor do Dataloader so

    • batchsize=1, que indica o tamanho do batch (default 1)
    • shuffle=true, que embaralha as observaes (antes de quebrar em lotes) a cada poca (default true).
    • partial=false, que pode ou no descartar o ltimo batch se for menor do que o batchsize desejado (default false).

  • Tambm podemos combinar isso com o IterTools.ncycle para economizar um loop (isso est exemplificado no caderno sobre identificao de dgitos).

  • Vamos ver o efeito dessas opes em uma sequncia de dados.
